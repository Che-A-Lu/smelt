# 空间交互规范 v1.0

> 这不是需求文档。这是**实现标准**。每一条都精确到帧、像素、毫秒。
> 任何开发者（人或 AI）拿到这份文档，应该能独立实现出对味的交互。
> 做不到的，是这份规范没写清楚，不是实现者的问题。

---

## 零、核心原则

**空间不建功能，空间只渲染卡片和传递事件。**

空间是一个极薄的渲染层。它的全部职责：
1. 渲染卡片在画布上的位置和状态
2. 把用户手势翻译成事件（拖拽、点击、双击、右键）
3. 把卡片之间的碰撞/邻近关系翻译成事件（拖到球上、拖到包上、拖到工作台上）

空间不做的事情：
- 不管理卡片数据（那是 storage 的事）
- 不调用 AI（那是 ai feature 的事）
- 不处理导入导出逻辑（那是 import/export feature 的事）
- 不做业务决策

**一张卡片就是一个 DOM 节点。不存在"临时 UI 元素"。** 对话框是卡片。流式输出是卡片。右键菜单是卡片（定位到指针位置、自动消失）。

## 一、帧预算

所有动画必须在以下预算内完成。任何单帧渲染超过 16ms 即为 bug。

| 动画类型 | 时长 | 缓动函数 |
|---------|------|---------|
| 卡片出现（fade in） | 150ms | `cubic-bezier(0, 0, 0.2, 1)` — 即 ease-out |
| 卡片消失（fade out） | 100ms | `cubic-bezier(0.4, 0, 1, 1)` — 即 ease-in |
| 位置过渡（snap to grid） | 200ms | spring: stiffness 300, damping 28 |
| 拖拽跟随延迟 | 2 帧 (~33ms) | 线性插值，每帧追上当前位置的 40% |
| 状态切换（颜色/边框/透明度） | 150ms | ease-out |
| 工作台打开/关闭 | 250ms | spring: stiffness 200, damping 24 |
| 卡片入包/出包 | 200ms | spring: stiffness 350, damping 30 |
| Toast 出现/消失 | 200ms fade + 8px 上移 | ease-out / ease-in |

**帧率硬要求**：拖拽和滚动期间 60fps 不可妥协。任何导致丢帧的计算必须移到 requestIdleCallback 或 Web Worker。

## 二、卡片物理

### 2.1 静止状态

```
宽高比 5:7（扑克牌比例）
默认尺寸：150 × 210 px

外观层级（从底到顶）：
  1. 卡片底色 #ffffff
  2. 左侧 3px 状态线（颜色见状态表）
  3. 左上角类型色块（6×12px 矩形，颜色见类型表）
  4. 卡片名称（10px 字号，色块右侧，单行截断）
  5. 内容预览区（9px 字号，#9ca3af，最多 5 行）

边框：1px solid #e5e7eb
圆角：6px
无阴影
```

### 2.2 状态线与卡片状态

| 状态 | 状态线颜色 | 卡片整体 | 含义 |
|------|----------|---------|------|
| idle | #e5e7eb (灰) | 正常 | 默认 |
| active | #22c55e (绿) | 正常 | 正在被 AI 使用 |
| generating | #3b82f6 (蓝) | 卡片 60% 透明度，脉冲呼吸 | AI 正在生成这张卡片 |
| interrupted | #f59e0b (橙) | 卡片边框变橙 | 生成被中断 |
| orphaned | #d1d5db (浅灰) | 45% 透明度 | 引用的球已被删除 |
| selected | — | 边框变为 2px #1a1a2e | 被选中 |

**generating 状态呼吸动画**：
```css
@keyframes cardBreathe {
  0%, 100% { opacity: 0.6; }
  50%      { opacity: 0.85; }
}
/* duration: 1200ms, easing: ease-in-out, iteration: infinite */
```

### 2.3 拖拽物理

**阶段一：拾起（PointerDown → 前 3 帧）**

| 帧 | 卡片行为 |
|----|---------|
| 第 1 帧 | 卡片 scale 1 → 1.05（瞬间，无过渡） |
| 第 2 帧 | 阴影出现：`filter: drop-shadow(0 4px 12px rgba(0,0,0,0.15))` |
| 第 3 帧 | 开始跟随指针。Z-index 提升到当前最高 + 1 |

这三帧内指针移动不跟随。这是故意的——防止误触时卡片抖动。

**阶段二：跟随（第 4 帧起）**

卡片位置每帧更新。不是直接贴指针，而是**每帧向指针当前位置移动剩余距离的 40%**。这产生一个 2-3 帧的视觉延迟，像物理拖拽。

```
card.x += (pointer.x - card.x) * 0.4
card.y += (pointer.y - card.y) * 0.4
```

**阶段三：松手（PointerUp）**

1. 卡片 scale 1.05 → 1.0（200ms spring）
2. 阴影消失（150ms ease-out）
3. 检测落点：
   - 落在一张球上（距离 < 60px）→ 触发 ball-dropped 事件
   - 落在一个包上（距离 < PACK_RADIUS）→ 触发 pack-dropped 事件
   - 落在一个工作台上 → 触发 workbench-dropped 事件
   - 落在空白区 → 卡片以 spring 动画滑到最终位置
4. Z-index 恢复到正常

**扔出（Throw）**

松手时如果指针在移动（最近 3 帧平均速度 > 2px/帧），卡片继续以该速度惯性滑行，每帧速度衰减 × 0.92，直到速度 < 0.1px/帧 时弹簧吸附到最终位置。

```
velocity = avg(last 3 pointer deltas)
while |velocity| > 0.1:
  card.position += velocity
  velocity *= 0.92
```

### 2.4 双击

双击卡片：卡片名称进入编辑模式。名称原地变为 `<input>`，自动聚焦并选中全部文本。按 Enter 或失焦保存，按 Escape 取消。

编辑模式过渡：input 宽度从卡片名称宽度扩展到 220px（150ms ease-out）。

## 三、卡片生命周期动画

### 3.1 创建

新卡片出现序列（总时长 200ms）：
1. 0ms：卡片在目标位置，scale(0.8)，opacity: 0
2. 0-50ms：opacity 0 → 1（瞬间不是渐变，是一帧跳到 1）
3. 0-200ms：scale 0.8 → 1.0（spring 动画，轻微 overshoot 到 1.03 然后回落）
4. 状态线在 200ms 后从透明渐变为对应颜色（再加 100ms）

### 3.2 删除

1. 卡片 scale 1.0 → 0.85（100ms ease-in）
2. 同时 opacity 1 → 0（100ms ease-in）
3. 动画完成后从 DOM 移除
4. 邻近卡片不做位置调整（卡片没有自动布局）

### 3.3 状态切换

卡片 status 变化时：
1. 状态线颜色 150ms ease-out 过渡到新颜色
2. 如果是 generating → idle：opacity 从呼吸态回到 1.0，状态线从蓝渐变到绿（200ms ease-out，因为蓝→绿变化明显要稍微慢一点让人看到"完成了"）
3. 如果是 → orphaned：opacity 1 → 0.45（150ms ease-out）

### 3.4 入包 / 出包

卡片移入包时：卡片向包位置移动（200ms spring stiffness 350），过程中 scale 1.0 → 0.7。到达包位置后包图标有一个轻微弹跳（scale 1 → 1.08 → 1，200ms）。

卡片从包取出时：卡片从包位置滑出到目标位置（200ms spring stiffness 350），scale 0.7 → 1.0。

## 四、临时工作台

工作台是一张特殊的卡片——"会话卡"。它和其他卡片完全一样：有名字、可拖、可关、可入包。唯一的区别是：**双击打开后，卡片旁出现一个对话浮窗。**

用户的世界里只有卡片。没有"球"这个视觉概念。球是 OPFS 里的数据实体，用户永远看不到它。

### 4.1 创建

点击工具栏"新建工作台"按钮 → 画布中央出现一张工作台卡片，初始名称为"新会话"。卡片以标准创建动画出现（scale 0.8 → 1.03 → 1.0，200ms spring）。

### 4.2 展开与折叠

- **双击工作台卡片** → 卡片旁弹出对话浮窗。卡片自身微微放大（scale 1 → 1.03，100ms），表示"已打开"
- **再次双击或点浮窗外空白处** → 浮窗收起，卡片恢复 scale 1.0

展开/折叠动画：浮窗从卡片右侧滑入/滑出（200ms spring stiffness 200 damping 24）。

浮窗打开时拖拽工作台卡片 → 浮窗跟随移动（无延迟，直接绑定位置）。

### 4.3 内部架构：区（Zone）插槽系统

工作台浮窗内部不写死任何布局。它只是一个**区的容器**——从上到下依次渲染已注册的区组件。每个区是一个独立文件。加新区=新建一个文件+在区列表中加一行。

**区列表（`workbenchZones`）：**

```typescript
// 每个区是一个对象，有 id、标题、默认折叠态、渲染组件
const workbenchZones = [
  { id: "context",   title: "上下文顺序",    defaultOpen: false, Component: ContextZone },
  { id: "files",     title: "AI 临时文件",    defaultOpen: true,  Component: FilesZone },
  { id: "tray",      title: "卡片陈列",       defaultOpen: true,  Component: TrayZone },
  { id: "history",   title: "对话历史",       defaultOpen: true,  Component: HistoryZone },
];
```

**区组件的统一接口：**

```typescript
interface ZoneProps {
  sessionId: string;            // 此工作台的唯一 ID
  messages: WbMessage[];        // 对话消息列表（共享只读）
  onMessagesChange: (fn: (prev: WbMessage[]) => WbMessage[]) => void;  // 修改消息
  tempFiles: TempFile[];        // AI 产出的临时文件列表
  onTempFilesChange: (fn: (prev: TempFile[]) => TempFile[]) => void;
  trayCards: string[];          // 陈列区卡片 ID 列表
  onTrayCardsChange: (fn: (prev: string[]) => string[]) => void;
  contextOrder: ContextItem[];  // 上下文顺序
  onContextOrderChange: (fn: (prev: ContextItem[]) => ContextItem[]) => void;
  onCreateCard: (title: string, fileData?: { name: string; content: string }) => void;
  isBusy: boolean;              // AI 是否正在推理
}
```

每个区接收完整的工作台状态。它只读自己关心的字段、只写自己能改的字段。区之间不直接通信——通过共享状态修改。

**折叠行为：**

每个区有一个标题栏，点击折叠/展开。折叠态只显示标题 + 角标数字。例如：
- "AI 临时文件 **3**"（3 个待处理文件）
- "上下文顺序 **5**"（5 个项目）

### 4.4 各区详细规范

#### 4.4.1 上下文顺序区（ContextZone）

位置：最顶部。默认折叠。

内容：一个可拖拽排序的标签列表。每个标签显示来源图标 + 名称。

来源：
- 用户在对话历史区勾选的消息段落 → 自动作为标签追加到列表末尾
- 用户把卡片拖到工作台上 → 自动作为标签追加到列表末尾

操作：
- 上下拖拽排序（其他标签让位，spring 200ms）
- 右键标签 → "移除"（不删原卡片/消息，只是移出上下文）
- 列表底部的顺序 = AI 看到 messages 数组的顺序

#### 4.4.2 AI 临时文件区（FilesZone）

位置：第二区。默认展开。

内容：每个 AI 产出的临时文件一行。

```
┌──────┬──────────────────────────┬────────┬──────────────┐
│ 色块  │ 文件名                    │ 大小    │    [建卡]     │
├──────┼──────────────────────────┼────────┼──────────────┤
│  MD  │ 分析报告.md               │ 12KB   │   建卡       │
│  PY  │ cleanup.py                │  2KB   │   建卡       │
│ CSV  │ 临时数据.csv              │  8KB   │   建卡       │
└──────┴──────────────────────────┴────────┴──────────────┘
```

行为：
- AI 每次 tool call 产出文件 → workbench 收到 → `onTempFilesChange` 追加 → FilesZone 自动更新
- 点"建卡" → 调用 `onCreateCard(title, fileData)` → 画布上出现新卡片 + 文件从临时列表中移除
- 清除工作台时 → 所有未建卡的临时文件删除
- 没有临时文件时 → 折叠态角标为 0，不显示

#### 4.4.3 卡片陈列区（TrayZone）

位置：第三区。默认展开。

内容：水平排列的卡片迷你缩略图（80×112px，内容预览前 2 行）。

来源：
- 在工作台内点"新建卡片" → 卡出现在陈列区
- 从画布拖一张卡到工作台卡片上 → 卡出现在陈列区（原卡片位置不变，但陈列区持有引用）

操作：
- 从陈列区拖出 → 卡回到画布（如果是从画布借入的，归还原位；如果是新建的，放置在拖出位置）
- 双击缩略图 → 卡片的名称进入编辑模式
- 清除工作台 → 新建的卡和借入的卡全部归还画布

#### 4.4.4 对话历史区（HistoryZone）

位置：底部，占剩余空间。默认展开。**此区是唯一不可折叠的区。**

内容：当前已有实现——每条消息一个段落，用户消息和 AI 回复交替排列。AI 回复可勾选段落、可折叠。新增功能：

- 每条 AI 回复末尾有一个小按钮 `[+]` → 点击后该回复中选中的段落追加到上下文顺序区
- 可拖拽排序消息（与现有实现一致）

### 4.5 区之间的数据流

```
用户在 HistoryZone 勾选段落 → onMessagesChange
    ↓
Workbench 检测 checked 的消息 → 追加到 contextOrder
    ↓
ContextZone 自动渲染新标签

AI 产出文件 → Workbench 收到 stream done 事件
    ↓
追加 tempFiles → onTempFilesChange
    ↓
FilesZone 自动渲染新行

用户点 FilesZone 里的"建卡" → onCreateCard
    ↓
Workbench 调 storage.createCard → 卡片出现在画布 → 追加到 trayCards
    ↓
TrayZone 自动渲染新缩略图

用户拖拽 ContextZone 标签排序 → onContextOrderChange
    ↓
下次 send() 时，contextOrder 的顺序 = AI messages 的顺序
```

### 4.6 工作台底部按钮

四个按钮不变：
- **[发给 AI]** — 读取 contextOrder 构建 messages 数组 → 调 AI → 流式输出追加到 HistoryZone
- **[新建卡片]** — 弹小输入框 → 标题 + 类型 → 创建卡片 → 加入 TrayZone
- **[归档全部]** — 全部对话 + 临时文件 + 上下文 → 生成一张"工作记录卡" → 清除工作台
- **[清除工作台]** — 删除临时文件、归还陈列区卡片、关闭浮窗

### 4.7 可扩展性：加一个新区的步骤

假设要给工作台加一个"笔记区"——用户可以随手写字、不发给 AI。

1. 新建 `src/ui/workbench/NotesZone.tsx`，实现 `ZoneProps` 接口
2. 在区列表里加一行：
```typescript
{ id: "notes", title: "随手笔记", defaultOpen: false, Component: NotesZone },
```
3. 完成。不需要改 Workbench.tsx 任何其他代码。

工作台不关心区做什么——它只负责渲染区列表、把共享状态传给每个区。

### 4.5 多个工作台

画布上可以同时存在多个工作台。每个工作台是独立的 AI 会话。不同工作台之间不共享上下文。

### 4.6 工作台卡片的外观

工作台卡片和普通卡片长得一样（150×210px 扑克牌，左侧状态线，左上角类型标识，名称，内容预览），但有两点区分：

- 类型色块固定为 #1a1a2e（黑色）——"会话卡"的标识色
- 内容预览区显示最近一条 AI 回复的前 5 行摘要（灰色小字，和普通卡片一致）

工作台卡片在 AI 推理时进入 generating 状态——状态线变蓝、卡片呼吸动画。推理完成变回 idle（状态线变灰）。普通卡片 currently being generated by AI 也是 generating 状态——外观完全相同。

### 4.7 上下文区与工具区：两种加入方式

工作台内有两种"给 AI 的东西"，必须分两个区：

| 区 | 给 AI 的什么 | AI 拿它做什么 | 加入后在哪里 |
|----|------------|-------------|------------|
| 上下文区 | **数据**：文件、报告、参考文本 | 阅读、理解、作为回答依据 | 变成 messages 数组里的 content |
| 工具区 | **工具**：脚本卡、插件卡、功能卡 | 调用、执行、拿结果 | 变成 API 请求里的 tools 定义 |

两张加入方式，同时支持：

**方式 A：拖拽。** 从画布拖一张卡到工作台浮窗→浮窗亮起两个接收区高亮（上下文区高亮蓝色、工具区高亮紫色）→用户松手时指向哪个区，卡片就加入哪个区。松手位置不在两个区上→不加入，卡片归位。

**方式 B：选择面板。** 点击上下文区或工具区的 `[+]` 按钮→弹出画布卡片选择器（一个浮层列表，列出画布上所有卡片）→勾选要加入的卡片→点"确定"。多选支持。

**加入是引用，不是移动。** 卡片加入工作台后，原卡片仍然在画布原位置。AI 读取的是卡片数据的副本。工作台清除时引用解除，原卡片不变。

**工作台是沙箱。** AI 只能看到工作台里有的卡片。画布上有 50 张卡，工作台上下文区只有 3 张、工具区只有 1 张——AI 不知道另外 46 张存在。用户放进来的，AI 才能看见。

### 4.8 工具声明：每条 tool 必须告诉 AI 三件事

云端模型收到 tools 定义，只是一段 JSON。它不知道自己在浏览器沙箱里。**我们能控制的不是 AI 做了啥，是 AI 收到的那段声明写了啥。**

每条 tool 声明必须包含：

1. **能做什么** — 功能描述（不是技术描述："读取卡片内容"而非"调用 card.read endpoint"）
2. **不能做什么** — 直接列出限制（"不能修改卡片，只能读取"）
3. **需不需要确认** — 明确告知（"此操作只读，无需用户确认" 或 "此操作会修改数据，需用户授权"）

示例声明：

```
tool: card.read
  功能：读取指定卡片的内容和元信息（标签、创建时间、文件类型）
  限制：只读操作，不能修改卡片。不能访问工作台外的卡片
  确认：不需要
  参数：
    - cardId (必需): 卡片在上下文区或工具区中显示的名称

tool: card.update
  功能：修改指定卡片的内容
  限制：只能修改已加入此工作台的卡片，不能改工作台外的卡片
  确认：需要用户手动批准
  参数：
    - cardId (必需)
    - newContent (必需): 新内容
```

声明不靠 AI 信任——靠 Worker 沙箱在执行层硬拦截。声明告诉 AI 边界在哪，沙箱确保 AI 跨不过边界。

### 4.9 内置工具集

空间出厂携带一套最小工具。用户不装任何工具卡就能完成基础工作。

**读写类（只读，无需确认）：**

| tool | 功能 | 限制 |
|------|------|------|
| `card.read` | 读取一张卡片的内容和元信息 | 只能读工作台内的卡片 |
| `card.list` | 列出当前工作台上下文区和工具区的所有卡片 | — |
| `card.search` | 按标签或名称搜索画布上的卡片（不读内容——只返回卡片名+标签） | 不能搜工作台外的卡片内容 |

**写入类（需用户手动确认，或处于自动模式下免确认）：**

| tool | 功能 | 确认 |
|------|------|:---:|
| `card.create` | 在工作台的 AI 临时文件区创建新卡片 | 需要（自动模式下免） |
| `card.update` | 修改一张卡片的内容 | 需要（自动模式下免） |
| `card.tag` | 给卡片添加或移除标签 | 不需要（纯标签操作） |

**危险类（AI 不能调用，只能口头建议）：**

| tool | AI 能做 | 用户必须 |
|------|--------|---------|
| `card.delete` | 说"建议你删掉这张卡" | 手动右键删除 |

**执行类：**

| tool | 功能 | 限制 |
|------|------|------|
| `script.run` | 执行工具区中的脚本卡 | 在 Worker 沙箱里跑。无网络、无真实文件系统 |

工具区里每张脚本卡自带一段作者写的"自述声明"（卡片 manifest 的 `description` 字段）。这段声明直接注入到 `script.run` 的 tools 描述里，告诉 AI："这张脚本叫什么、做什么、需要什么输入、输出什么"。

### 4.10 工具执行模式：自动 / 手动

用户在设置面板中选择，全局生效，每个工作台可单独覆盖。

**手动模式（默认）：**

AI 每次调写入类 tool（`card.create`、`card.update`、`script.run`）→ Workbench 弹出确认条："AI 想要执行 card.update，修改卡片'分析报告'。允许吗？" → [允许] [拒绝] [本次会话全部允许]。

只读类 tool（`card.read`、`card.list`、`card.search`）不弹确认。

**自动模式：**

AI 调任何 tool 直接执行，不弹确认。状态条上显示最近一次 tool call："AI 刚才调用了 script.run → cleanup.py"。

用户在任意时刻可以从自动切回手动，反之亦然。切换立即生效。

### 4.11 三重沙箱

用户不需要理解这些——但空间必须在每层都设墙。

| 层 | 墙 | 防什么 |
|----|----|--------|
| 浏览器 | 无真实 shell、无进程、无 `/usr/bin` | AI 不可能执行系统命令 |
| Web Worker | 无 DOM、无网络、只有我们给的 API | 工具卡脚本跑不出去 |
| 工作台 | AI 只看到被加入工作台的卡片 | 画布上 50 张卡，工作台里 3 张，AI 不知道另外 47 张 |

**用户想做的事 vs 空间怎么做到：**

| 用户想 | 传统工具 | 空间替代 |
|--------|---------|---------|
| 分析 CSV | `pandas` | AI 直接读文本推理，或内置 JS 数据处理 |
| 批量改名 | shell `mv` | `card.update` 逐张改 |
| 跑 Python 脚本 | `python script.py` | 脚本卡放工具区，AI 调 `script.run` |
| pip install | `pip install xxx` | 做不到。但如果功能重要，就会有人做成一张工具卡 |
| 调外部 API | `curl` | 暂不允许。未来可选开放"网络请求"工具卡 |

上限不在技术——在生态。空间只有内置工具时上限低。但工具区能插拔外部工具卡后，任何人可以发布"数据分析.card"、"图片处理.card"、"记忆搜索.card"——上限在社区手里。

## 五、画布

### 6.1 平移

- 空白区域 PointerDown → 拖拽平移（无延迟，直接跟随）
- 平移时卡片不重渲染（只改容器 transform）
- 平移惯性：松手时若指针仍在移动，画布以当前速度 × 0.95 继续滑行直到停止

### 6.2 缩放

- 滚轮缩放：以指针位置为中心缩放，步进 0.1，范围 [0.2, 3.0]
- 缩放时背景网格大小同步更新（间距 = 40 × scale px）
- 双指手势（触控板/触摸屏）：双指捏合缩放 + 双指滑动平移

### 6.3 背景

```
底色：#fafbfc
网格：radial-gradient(circle, #e5e7eb 1px, transparent 1px)
网格间距：40 × scale px（跟随缩放同步变化）
网格原点：跟随画布偏移（view.x × scale, view.y × scale）
```

### 6.4 拖入文件

浏览器外拖入文件时：

1. 文件进入浏览器窗口 → 全屏覆盖层出现
   - 虚线边框 2px dashed #1a1a2e
   - 中央提示文字："释放以导入"
   - 覆盖层 opacity 0 → 1（100ms ease-out）

2. 文件离开浏览器窗口 → 覆盖层消失（100ms ease-in）

3. 松手释放：
   - .card 文件 → 触发导入流程（ImportDialog 弹出，也是卡片）
   - 普通文件 → 创建卡片，卡片出现在释放位置
   - 单文件：卡片出现后自动进入编辑模式（名称选中）
   - 多文件：Toast 提示"已导入 N 个文件"

## 六、AI 流式输出

### 7.1 流式输出的位置

流式输出**不创建独立组件**。它在工作台浮窗的对话历史区内追加一条新 AI 回复段落。

### 7.2 状态流转

```
idle
  → thinking（用户点"发给 AI"后立即进入）
    → streaming（收到第一个 content chunk 时）
      → done（收到 done 信号）
      → error（收到 error 或网络异常）
    → interrupted（用户点中断）
```

### 7.3 各阶段视觉

**thinking**：
- 显示一个 16px 直径的双环圆圈，中心一个更小的实心圆，以脉冲动画呼吸（1000ms ease-in-out 循环）
- 旁边文字："思考中..."
- thinking 阶段的真实思考 trace 存储在内存，不展示，只在 done 后作为可展开段落出现

**streaming**：
- 文字逐字出现（不是逐 token，是 CSS 逐字符 typing 动画或者直接用 requestAnimationFrame 追加 content）
- 追加速度：真实流式速度（不人为减速）
- 当前段落末尾有一个闪烁光标 `|`（CSS blink 动画，500ms 间隔）

**done**：
- 闪烁光标消失
- AI 回复段落变为可勾选、可折叠
- 同时把 thinking trace 作为一个折叠区附加在此段落下方（标题："思考过程"）
- 如果 AI 产出了文件（tool call 结果），在 AI 文件区同时出现新条目

**error**：
- AI 回复段落底部显示红色错误消息
- 段落保留已接收的内容（不丢失）
- 段落旁有一个"重试"按钮

**interrupted**：
- 当前内容保留
- 段落底部显示橙色标签："已中断"
- 已接收内容和临时文件保留

### 7.4 中断

用户点工作台浮窗底部的 [中断] 按钮：
1. abort() API 请求
2. 已接收的流式内容保留在对话区
3. thinking trace 保留
4. 状态切换为 interrupted
5. 不会自动创建新卡片（只有 done 时才创建）

## 十二、存储与持久化

### 12.1 存储后端

空间所有数据（卡片索引、球数据、设置、API key）存储在浏览器 OPFS（Origin Private File System）中。OPFS 是浏览器为每个网站分配的私有文件系统，其他网站无法访问。

### 12.2 持久化：防止浏览器自动清理

**浏览器有权不经用户同意清理 OPFS 数据。** 触发条件：磁盘空间紧张、用户长时间未访问、系统级存储优化。

空间在**首次加载时**静默调用 `navigator.storage.persist()`：

- 浏览器弹出原生提示："此网站想要永久存储数据"
- 用户同意 → 数据被自动清理的概率降到极低
- 用户拒绝或忽略 → 静默失败

`navigator.storage.persisted()` 返回当前持久化状态（boolean）。

### 12.3 持久化状态显示

画布左下角显示存储状态条：

```
已用 12MB  ·  持久化 ✓
```

若持久化未获得：

```
已用 12MB  ·  持久化 ✗  数据可能被清理
```

后者用橙色文字（`#f59e0b`），右箭头 → 点击跳转设置面板。

### 12.4 设置面板中的持久化入口

设置面板顶部显示持久化状态。若未获得，旁边有一个按钮"申请永久存储"。点击→再次调用 `persist()` → 更新状态。

若浏览器拒绝时无原生提示（静默失败），按钮旁显示提示文字："浏览器不支持此功能。请定期导出 .card 备份。"

### 12.5 API Key 的保护

API key 在写入 OPFS 前使用 AES-GCM 加密（`platform/settings.ts`）。key 与卡片数据共享同一份持久化保护。

若持久化未获得，API key 与卡片数据面临同样的清理风险。**唯一可靠的防护是拿到 `persist()` 权限。**

附加保险：用户可在设置面板中将所有配置（含 API key）加密导出为一张 .card。浏览器清理后，拖入这张 .card 即可恢复全部密钥。

### 12.6 存储用量显示

画布左下角的存储条实时显示已用量和配额。配额通过 `navigator.storage.estimate()` 获取。到达 80% → 条变黄。到达 95% → 条变红 + Toast 提示"存储空间不足，请清理临时文件或导出旧卡片"。

### 12.7 清理策略

- 工作台清除时 → 删除该工作台的所有临时文件
- 卡片删除时 → 其关联的球数据（若无其他卡片引用）一并清理
- 设置面板提供"清除所有临时文件"按钮
- 从不自动删除用户的卡片和球——只删除明确标记为临时文件的数据

### 12.8 终极防线

不管 `persist()` 是否成功，用户应养成习惯：**重要的东西打成 .card，存到自己的硬盘里。** 空间丢了就丢了，.card 拖回来五分钟恢复。

## 十三、响应式与性能

### 13.1 卡片数量

- 0-50 张：所有动画和交互全开
- 50-200 张：禁用卡片间碰撞检测，拖拽时省略阴影
- 200+ 张：进入"地图模式"——卡片缩小为色点，只显示名称，不支持拖拽

### 13.2 渲染策略

- 画布外（屏幕不可见区域）的卡片：跳过渲染（保留 DOM 节点但 `display: none`，由 IntersectionObserver 管理）
- 流式输出文字：使用 requestAnimationFrame 批量更新 DOM，不在每个 chunk 都触发重排

### 13.3 内存

- 卡片预览缓存（previewCache）上限 100 条，LRU 淘汰
- 工作台关闭时清除所有临时文件和该工作台关联的缓存

## 十五、导入器：信任边界

### 14.1 定位

导入器是 .card 进入空间的**唯一入口**。它不是简单的"打开文件"——它是一个检查、验证、解密、选择性导入的严谨过程。

用户拖入一张 .card → 不是直接导入，而是先弹出导入器界面。导入完成后的卡片才出现在画布上。

### 14.2 导入流程（四步）

**第一步：解包**

读取 .card 的 ZIP 结构，解析 manifest.json。如果文件损坏或不是有效的 .card → 显示错误，终止。

**第二步：安全检查**

此时卡片数据尚未进入空间，是在沙箱态下检查：

| 检查项 | 说明 |
|--------|------|
| manifest 完整性 | 必需字段齐全（id、label、version） |
| 文件清单一致性 | manifest 中记录的文件列表与 ZIP 内的实际文件匹配 |
| 文件大小校验 | 逐个文件对比 manifest 中记录的 size 与实际大小 |
| SHA-256 校验 | 若 manifest 含 checksums，逐一校验文件哈希 |
| 无恶意文件 | 检测 artifact 中是否包含 .exe、.dll、.sh、.bat 等可执行文件（如有则警告） |

任何一项不通过 → 显示具体是哪项失败 + 风险等级 + 是否仍然继续（用户自担风险）。

**第三步：加密与签名状态**

界面上清晰显示：

```
 签名状态：已签名 · Dalu (ECDSA P-256)
           签名验证：通过 ✓
           签名时间：2026-07-17 14:30 UTC

 加密状态：已加密 · AES-256-GCM
           需要密码才能导入
           [输入密码: ____________]
```

- 签名验证通过 → 绿色 ✓ + 签名者身份
- 签名验证失败 → 红色 ✗ + "内容可能被篡改"
- 未签名 → 灰色 "—" + "此卡片未签名，来源不可验证"
- 已加密 → 必须输入密码才能进入下一步
- 未加密 → 跳过

**第四步：选择导入内容**

用户看到 .card 内的完整内容清单，决定哪些导入、哪些不导入：

```
┌──────────────────────────────────────────────┐
│  导入内容选择                                  │
│                                              │
│  卡片名称：年度分析报告                        │
│  作者：Dalu    版本：1.0.0                     │
│                                              │
│  ├── manifest.json        2KB   ✓ 必需       │
│  ├── process.jsonl       18KB   ☑ 过程记录   │
│  └── artifacts/                             │
│      ├── ☑ data.xlsx     45KB   数据文件     │
│      ├── ☑ report.md     12KB   分析报告     │
│      ├── ☐ notes.md       3KB   随手笔记     │
│      └── ☐ temp/          8KB   AI 临时文件  │
│                                              │
│  [全选] [仅导入产物] [仅导入过程]              │
│  [确认导入]                                   │
└──────────────────────────────────────────────┘
```

- 默认：全部打勾（用户直接点"确认导入"就行）
- "仅导入产物"：只勾 artifacts 里的文件，不导入 process.jsonl
- "仅导入过程"：只勾 process.jsonl，不导入产物文件
- 临时文件默认不打勾（AI 的中间产物通常不需要导入）
- 用户可逐项勾选/取消

### 14.3 导入完成

点"确认导入"后：

1. 卡片以标准创建动画出现在画布中央（scale 弹出）
2. 所有 artifacts 文件写入 OPFS（球）
3. process.jsonl（如导入）写入 OPFS，关联到卡片
4. 若 .card 是"主题卡"（manifest.type === "theme"），除导入卡片外，还弹出"是否应用此主题？"
5. Toast 提示"已导入：年度分析报告"

### 14.4 导入器的独立性

导入器是空间的独立模块（`features/import/`），不混在 Space.tsx 里。它自己管理自己的四步状态机（unpack → verify → decrypt/auth → select）。导入完成后把结果（卡片数据 + 球数据）交给 storage 层，UI 层通过 storage 的回调感知到新卡片出现。

### 14.5 安全原则

- **导入器不做猜测。** 签名验证失败不一笔带过——明确告诉用户"可能被篡改"
- **可执行文件不静默导入。** 检测到 .exe/.sh/.bat 等必须弹警告
- **加密的 .card 必须解密后才能看到内容清单。** 不解密时只显示 manifest 的公开字段
- **导入后的卡片在空间里和其他卡片一样。** 没有"这个卡片来自外部"的特殊标记——一旦导入，它就是用户的资产

## 十六、可覆盖性：交互参数不是写死的

以上所有交互参数——时间、距离、比例、曲线——是**空间的出厂默认值**，不是硬编码常量。

一张 .card 可以是一个"主题卡"或"特效卡"。它不携带协作内容——它只携带交互参数的覆盖值。拖入空间后，空间读取其中的覆盖值，替换默认参数。拔掉这张卡，恢复默认。

### 14.1 覆盖机制

.card 的 `manifest.json` 里可以有一个可选字段 `interaction`。字段名对应本规范中的参数名。未覆盖的字段保持默认值。

```json
{
  "manifest": {
    "id": "theme-brutalist-001",
    "label": "粗野主义主题",
    "type": "theme",
    "version": "1.0.0",
    "author": "某个设计师"
  },
  "interaction": {
    "card": {
      "dragDelayFrames": 0,
      "dragFollowRatio": 1.0,
      "dragScaleOnPickup": 1.0,
      "dragShadow": "none",
      "throwDecay": 0.85,
      "snapSpringStiffness": 500,
      "snapSpringDamping": 40
    },
    "createAnimation": "none",
    "deleteAnimation": "cut"
  }
}
```

这张卡把拖拽改成了"即时跟随、零延迟、无阴影、硬弹到位"——粗野主义的感觉。另一张卡可以让卡片像羽毛一样飘（followRatio 0.1, decay 0.98）。

### 15.2 覆盖层级

```
全局默认值（本规范定义）
    ↓ 被覆盖
当前激活的主题卡（全局生效）
    ↓ 被覆盖
单张卡片的 self-interaction（只影响自己）
```

同一时间只有一张主题卡生效。先拖入的 A 主题生效，后拖入 B 主题→弹出选择"替换当前主题？"。

单张卡片可以在自己的 manifest 里带上 `interaction` 字段，只覆盖自己的交互参数。一个作者可能想让自己的卡片"特别重"——别人拖它的时候感觉吃力，这是一种表达。

### 15.3 可覆盖参数完整列表

#### 卡片拖拽

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `card.dragDelayFrames` | 3 | 拾起延迟帧数 |
| `card.dragFollowRatio` | 0.4 | 每帧追上指针的比例（0-1） |
| `card.dragScaleOnPickup` | 1.05 | 拾起时放大倍数 |
| `card.dragShadow` | `"0 4px 12px rgba(0,0,0,0.15)"` | 拖拽时阴影 CSS |
| `card.throwDecay` | 0.92 | 惯性衰减系数（0-1） |
| `card.minThrowVelocity` | 2 | 触发惯性投掷的最小速度（px/帧） |

#### 弹簧物理

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `spring.stiffness` | 300 | 弹簧硬度。越大越硬，越小越软 |
| `spring.damping` | 28 | 弹簧阻尼。越大越快停下来，越小越弹 |

不同感觉的例子：
- 默认弹簧（300/28）：干脆，有轻微 overshoot
- 软弹簧（150/15）：像果冻，弹好几下才停
- 硬弹簧（500/40）：一放就停，几乎不弹

#### 卡片生命周期

| 参数 | 默认值 | 可选值 |
|------|--------|--------|
| `createAnimation` | `"spring"` | `"spring"` / `"fade"` / `"none"` / `"drop"` |
| `deleteAnimation` | `"shrink"` | `"shrink"` / `"fade"` / `"none"` / `"explode"` |

- `drop`：卡片从上方掉下来（重力感）
- `explode`：卡片碎裂成粒子消失
- `fade`：纯淡入淡出

#### 工作台

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `workbench.openSpringStiffness` | 200 | 打开弹簧硬度 |
| `workbench.openSpringDamping` | 24 | 打开弹簧阻尼 |
| `workbench.panelWidth` | 360 | 浮窗宽度（px） |
| `workbench.showThinkingTrace` | true | 是否默认展开思考过程 |

#### 卡包

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `pack.expandSpringStiffness` | 250 | 展开弹簧硬度 |
| `pack.expandSpringDamping` | 24 | 展开弹簧阻尼 |
| `pack.cardInPackScale` | 0.7 | 卡片入包后的缩放比 |

#### 全局

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `global.backgroundColor` | `"#fafbfc"` | 画布底色 |
| `global.gridColor` | `"#e5e7eb"` | 网格点颜色 |
| `global.gridSpacing` | 40 | 网格间距（px） |
| `global.fontFamily` | `"system-ui, sans-serif"` | 全局字体 |
| `global.animationEnabled` | true | 关闭后所有动画变成 0ms |

### 15.4 安全边界

交互覆盖是**纯视觉的**——它只改变 CSS、动画参数、缓动曲线。它不能：
- 读其他卡片内容
- 访问 OPFS
- 发送网络请求
- 执行脚本

这不是权限限制——是"交互覆盖"这个机制本身的定义：它只是一组数字参数，不是代码。把它想象成 CSS 自定义属性——改颜色、改动画时长、改弹簧软硬——都是在渲染层的参数替换，没有执行任何逻辑。

**谁都可以写、谁都可以分享——不需要代码能力，改数字就行。**

## 十七、卡座：卡片通电的地方

### 16.1 核心原则

**卡片不分类型。** 一张卡片携带什么——数据、脚本、样式参数、音效——都由它的文件内容决定。导入器只识别文件格式，在角标上显示文件类型和文件名。不做任何"这是功能卡""那是资产卡"的判断。

**所有卡片默认不运行。** 导入后就是画布上的普通卡片，可以拖、可以入包、可以拖进工作台给 AI 看。

**运行与否取决于卡片在不在卡座上。** 不在卡座 = 静态数据卡。拖进卡座 = 通电，沙箱里跑起来。

### 16.2 卡座是什么

卡座是画布上一个**可见的区域**。和卡包、工作台并列——它们是画布上的三种基本容器：

| 容器 | 形态 | 作用 |
|------|------|------|
| 卡包 | 虚线框/展开条 | 收纳卡片，组织文件 |
| 工作台 | 卡片+浮窗 | 让 AI 看见卡片，协作对话 |
| 卡座 | **实体底座** | 给卡片通电，让它跑起来 |

卡座的外观：一个矩形底座区域，比卡片略大（约 180×250px），边框为实线（区别于卡包的虚线）。空卡座时显示浅灰色底+虚线内框，提示"拖卡片或卡包到此处以激活"。

### 16.3 同一张卡，三种用法

以一张 `cleanup.py` 脚本卡为例：

| 卡片在哪里 | 它在做什么 |
|-----------|-----------|
| 画布上，不在任何容器里 | 一张静态卡片。可以预览内容、改名、和其他卡片一样 |
| 拖进工作台 | AI 看到它。AI 理解这是一段 Python 脚本，可以把它当工具调用——对文件做清理操作 |
| 拖进卡座 | 通电。沙箱里跑起来。可能定时清理临时文件，可能注册一个右键菜单项 |

同一张卡。放哪里决定它干什么。**不是卡的类型决定行为——是用户的操作决定行为。**

### 16.4 卡座的操作

- **放入**：拖一张卡片到卡座上 → 状态线变绿（active）→ 卡片脚本在沙箱中启动
- **放入卡包**：拖一个卡包到卡座上 → 包内所有卡片一起激活 → 卡包整体显示为绿色状态线
- **取出**：从卡座拖出卡片 → 状态线变灰（idle）→ 脚本停止，所有注入的 UI 清理
- **多个卡座**：空间可以有多个卡座。每个卡座独立运行
- **卡座本身不可拖拽**（固定在画布上，类似卡包）

### 16.5 卡座的磁吸交互

卡座不能随便碰一下就激活——必须是**用户明确放入**才算。区分"经过"和"放入"的机制：

**接近阶段（卡片被拖拽，进入卡座周围 80px 范围）：**

| 距离 | 卡座反应 | 卡片反应 |
|------|---------|---------|
| > 80px | 无反应 | 正常拖拽 |
| 40-80px | 卡座边框微微变亮（#d1d5db → #9ca3af，150ms ease-out） | 无变化 |
| < 40px | 卡座边框高亮（→ #1a1a2e），内框从虚线变实线。底座轻微发光：`box-shadow: 0 0 20px rgba(26,26,46,0.08)` | 卡片微微减速——跟随比例从 40% 降到 25%，产生"被吸住了"的感觉 |

**松手判定：**

- 松手时卡片中心在卡座范围内 → **放入成功**。卡片以 150ms spring 吸附到卡座正中央，然后状态线变色（灰→绿），脚本启动
- 松手时卡片中心在卡座范围外 → **未放入**。卡片正常落位（松手阶段三的标准行为），卡座恢复默认外观
- 路过（拖拽穿过卡座区域但没松手）→ 卡座高亮随着卡片离开而消退（40-80px 阶段反向），**不激活**

**关键设计：**

- 判定只在松手（PointerUp）那一刻做。拖拽过程中无论离多近，都不算放入
- 磁吸是视觉暗示——卡片靠近时"感觉被拉过去"，但用户仍然可以拉走
- 卡座不吸附已经在卡座上的卡片（已激活的卡从卡座拖出时，不触发接近效果）

### 16.6 卡包拖入卡座

当某个功能需要多张卡协作（比如记忆库系统需要 5 张依赖卡），作者把它们打成一个卡包。用户把整个卡包拖进卡座：

- 卡包内的卡片全部激活，但**卡座只显示卡包的外观**——不是 5 张卡密密麻麻排开
- 卡包成为一个"通电中的功能模块"
- 用户可以打开卡包看里面有哪些卡在跑，但不能单独停用其中一张——它是一个整体
- 从卡座拖出卡包→全员停用

这解决了你之前说的"功能复杂时卡座会密密麻麻"的问题。卡包就是收纳容器——无论里面多少张卡，卡座上只看到一个包。

### 16.7 一个完整例子：记忆库系统

作者发布"长期记忆库.card"。导入后解出一个卡包，里面有：

```
记忆库系统（卡包）
├── memory_db.json      — 数据库结构定义
├── memory_search.py    — 搜索脚本
├── memory_store.py     — 存储脚本
├── memory_tool.md      — AI 工具定义（告诉 AI 怎么用）
├── memory_ui.js        — 记忆面板 UI
└── README.md           — 作者说明书
```

作者在 README 里写：

> **如何使用：**
> 1. 把「记忆库系统」卡包拖进卡座 → 记忆库启动，画布出现记忆面板
> 2. 把 `memory_tool.md` 拖进工作台 → AI 就能调用记忆库
> 3. 不用时，从卡座拖出卡包 → 记忆库停止

用户操作：
1. 导入 → 卡包出现在画布上
2. 卡包拖进卡座 → 状态线变绿 → 记忆面板出现在画布上
3. 打开工作台 → 把 `memory_tool.md` 拖进去 → AI 看见了，开始调用记忆库的搜索和存储能力
4. 工作结束 → 工作台归档 → 记忆库因为还在卡座上，继续运行
5. 关掉空间 → 下次打开，卡包还在卡座上，记忆库自动恢复

**空间的复杂度不增加。增加的只有卡片。**

### 16.8 卡座上的注入点

卡座上的脚本通过空间的注入接口添加 UI：

```
空间提供的注入点：
  - canvas.overlay    → 画布上的自由 DOM 层（特效、粒子、面板）
  - canvas.toolbar    → 顶部工具栏的按钮位
  - canvas.floating   → 画布上的可拖拽悬浮元素
  - card.contextMenu  → 卡片右键菜单的扩展项
```

脚本不直接操作 DOM。空间是中间层——脚本说"我要在工具栏加一个按钮"，空间渲染它。脚本说"我要在画布上加一个面板"，空间渲染它。

### 16.9 沙箱

卡座上的脚本在 Web Worker 中执行（复用 `features/sandbox/`）：

| 能 | 不能 |
|----|------|
| 通过注入点向空间添加 UI | 直接操作 DOM |
| 读取同一卡座内其他卡片的数据 | 读取卡座外其他卡片的内容 |
| 访问自己的 OPFS 命名空间 | 访问 OPFS 的其他部分 |
| 使用 Web Animations API | 发送网络请求 |
| 响应空间事件（点击、时间推进） | 访问 API key 或用户设置 |

### 16.10 卡座与工作台的关系

**工作台是用来创造卡片的。卡座是用来运行卡片的。**

同一个脚本——在工作台里，AI 把它当工具调用（对文件操作）。拖到卡座上，它自己跑（定时清理、注册按钮、显示面板）。

用户不用理解"激活""停用"这些抽象概念——他只需要看到：**卡片在卡座上 = 在发光 = 在工作。**
